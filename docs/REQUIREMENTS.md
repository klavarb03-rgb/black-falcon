## Загальні вимоги

### Платформа
- **Desktop додаток** (не веб-інтерфейс)
- **Операційні системи:**
  - ✅ Windows (обов'язково)
  - ✅ Mac (бажано)
  - ❌ Linux (не потрібно)

### Режими роботи
- **Offline mode:** Менеджер може працювати без інтернету
- **Синхронізація:** При появі з'єднання — автоматична синхронізація з сервером
- **Conflict resolution:** Timestamp-based (пізніша зміна перемагає)

### Архітектура
- **Client-Server:**
  - Desktop app (Electron) на комп'ютері кожного менеджера
  - Центральний сервер (Backend + PostgreSQL + Ollama)
  - Менеджери в різних містах (підключення через Internet)

## Ролі користувачів

### 1. Керівник (Superadmin)

**Права:**
- ✅ Всі права (повний контроль)
- ✅ Перегляд звітів по всій організації (перехресно)
- ✅ Створення бібліотеки псевдонімів (nickname groups)
- ✅ Голосові запити через Telegram (аналітика)
- ✅ Моніторинг активності всіх менеджерів

**Особливості:**
- ❌ Немає власного складу (не працює з МЦ безпосередньо)
- Роль — контроль і аналітика

### 2. Адміністратор

**Права:**
- ✅ Управління користувачами (створення/редагування/видалення)

- ✅ Передача МЦ іншим менеджерам
- ✅ Списання МЦ (втрачено/використано)
- ✅ Створення/редагування/видалення МЦ
- ✅ Створення комплектів (BOM)
- ✅ Перегляд звітів по підлеглим менеджерам
- ❌ Не може видаляти дані інших адміністраторів

### 3. Менеджер (звичайний користувач)

**Права:**
- ✅ Облік МЦ на своєму складі (додавання нових позицій)
- ✅ Передача МЦ іншим менеджерам

- ✅ Створення комплектів (BOM)
- ✅ Перегляд звітів тільки по своєму складу
- ❌ Не може коригувати існуючі МЦ (тільки додавати нові)
- ❌ Не може видаляти МЦ
- ❌ Не може бачити дані інших менеджерів

## Матеріальні цінності (МЦ)

### Атрибути МЦ

**Обов'язкові поля:**
- **Назва (повна)** — текст, до 500 символів
- **Статус** — dropdown: "Державне" або "Волонтерське" (обов'язкове!)
- **Кількість** — число (залишок на складі)
- **Одиниця виміру** — dropdown: Штуки / Кілограми / Літри / Комплекти

**Необов'язкові поля:**
- **Серійний номер** — текст, до 255 символів
- **Вартість** — число, валюта: грн (необов'язкове поле!)
- **Відповідальний (хто отримав)** — прив'язка до користувача
- **Примітки** — текст + можливість прикріпити фото (декілька)
- **Донор** — вибір зі списку донорів (вказується при списанні, в примітках при отриманні)

### Додавання МЦ

**Способи:**
1. **Вручну** — форма в Desktop app
2. **Баркод/QR-код** — сканування через камеру
3. **Фото накладної** — через Telegram Import Bot (AI розпізнавання)
4. **Excel файл** — через Telegram Import Bot (AI парсинг)

**Workflow (вручну):**
```
1. Менеджер відкриває форму "Додати МЦ"
2. Заповнює поля:
   - Назва (обов'язково)
   - Статус (обов'язково: Державне / Волонтерське)
   - Кількість (обов'язково)
   - Одиниця виміру (обов'язково)
   - Серійний номер (опціонально)
   - Вартість (опціонально)
   - Примітки (опціонально)
   - Фото (опціонально)
3. Натискає "Зберегти"
4. МЦ додається на склад менеджера
5. Якщо online → синхронізація з сервером
6. Якщо offline → додається в локальну SQLite, синхронізація пізніше
```

## Операції з МЦ

### 1. Передача МЦ іншому менеджеру

**Сценарій:**
Менеджер А передає МЦ Менеджеру Б (в іншому місті).

**Workflow:**
```
1. Менеджер А відкриває МЦ зі свого складу
2. Натискає "Передати"
3. Обирає:
   - Кому (вибір менеджера зі списку)
   - Кількість (скільки передати)
   - Примітки (опціонально)
4. Підтверджує
5. МЦ -X у Менеджера А
6. МЦ +X у Менеджера Б
7. Notification для Менеджера Б (в Desktop app): "Ви отримали МЦ від [Менеджер А]"
8. Історія переміщень зберігається (хто → кому → коли → скільки)
```

**Документ:**
- Генерується PDF "Накладна на передачу" (автоматично)
- Містить: МЦ, кількість, від кого, кому, дата/час

### 2. Списання МЦ (втрачено)

**Сценарій:**
МЦ пошкоджене/загублене/вкрадене → потрібно списати.

**Workflow:**
```
1. Менеджер відкриває МЦ
2. Натискає "Списати (втрачено)"
3. Вказує:
   - Кількість (скільки списати)
   - Причина (текст: пошкоджено / загублено / вкрадено)
   - Донор (якщо треба звітуватись) — вибір зі списку
   - Примітки (опціонально)
4. Підтверджує
5. МЦ -X на складі
6. Запис в історії: "Списано (втрачено)"
```

**Якщо вказано донора:**
- Генерується звіт для донора (PDF/Excel)
- Можливість відправити email або роздрукувати

### 3. Списання МЦ (використано)

**Сценарій:**
МЦ передано кінцевому споживачу або використано за призначенням.

**Workflow:**
```
1. Менеджер відкриває МЦ
2. Натискає "Списати (використано)"
3. Вказує:
   - Кількість (скільки списати)
   - Кому (кінцевий споживач) — текст
   - Донор (якщо треба звітуватись) — вибір зі списку
   - Примітки (опціонально)
4. Підтверджує
5. МЦ -X на складі
6. Запис в історії: "Списано (використано)"
```

**Якщо вказано донора:**
- Генерується звіт для донора (PDF/Excel)

## Комплекти (BOM)

### Концепція

**Комплект (Bill of Materials)** — це набір окремих МЦ, які разом складають один виріб.

**Приклад:**
```
Комплект "Комп'ютер" складається з:
- Системний блок (1 шт)
- Монітор (1 шт)
- Клавіатура (1 шт)
- Миша (1 шт)
- Шнур живлення (1 шт)
```

### Створення шаблону комплекту

**Хто може:**
- Менеджер
- Адміністратор

**Workflow:**
```
1. Менеджер відкриває розділ "Комплекти"
2. Натискає "Створити шаблон комплекту"
3. Вказує:
   - Назва комплекту (наприклад, "Комп'ютер")
   - Додає компоненти (вибір зі свого складу):
     - МЦ (вибір зі списку)
     - Кількість (скільки потрібно)
4. Зберігає шаблон
```

**Особливості:**
- Шаблони комплектів — **не загальні** для всієї системи
- Кожен менеджер/адміністратор створює свої шаблони

### Збірка комплекту

**Сценарій:**
На складі є всі компоненти → менеджер збирає комплект.

**Workflow:**
```
1. Менеджер відкриває шаблон комплекту
2. Система перевіряє:
   - Чи є всі компоненти на складі?
   - Чи достатньо кількості?
3. Якщо так → показує "Можна зібрати X комплектів"
4. Менеджер натискає "Зібрати комплект"
5. Система:
   - МЦ-компоненти -1 (списуються зі складу)
   - Комплект +1 (з'являється на складі як окремий МЦ)
6. Історія зберігає: "Зібрано з [список компонентів]"
```

**Важливо:**
- Після збірки компоненти **зникають** зі складу
- З'являється **комплект** (як окремий МЦ)

### Розбирання комплекту

**Сценарій:**
Потрібно розібрати комплект назад на компоненти (рідкісний випадок).

**Workflow:**
```
1. Менеджер відкриває комплект
2. Натискає "Розібрати комплект"
3. Підтверджує
4. Система:
   - Комплект -1 (зникає зі складу)
   - МЦ-компоненти +1 (повертаються на склад)
5. Історія зберігає: "Розібрано на [список компонентів]"
```

**Особливості:**
- Можливість є, але **на практиці не використовується**

### Передача комплекту

**Сценарій:**
Комплект передається цілим (не окремими компонентами).

**Workflow:**
```
1. Менеджер А відкриває комплект
2. Натискає "Передати"
3. Обирає Менеджера Б
4. Комплект -1 у А
5. Комплект +1 у Б
```

## Групування МЦ

### Концепція

Менеджер може **групувати МЦ** для зручної навігації на своєму складі.

**Приклад структури:**
```
Обладнання (рівень 1)
└── Медичне (рівень 2)
    └── Діагностичне (рівень 3)
        └── Портативне (рівень 4)
            └── [МЦ тут]
```

### Вимоги

**Вкладеність:**
- До **4 рівнів** максимум

**Назви груп:**
- Менеджер **сам вводить** назви (без фіксованого списку)

**Один МЦ:**
- ❌ Не може бути в декількох групах одночасно
- Одна МЦ = одна група

### Workflow

**Створення групи:**
```
1. Менеджер відкриває розділ "Склад"
2. Натискає "Створити групу"
3. Вказує:
   - Назва групи (довільна)
   - Батьківська група (якщо це підгрупа)
4. Зберігає
```

**Переміщення МЦ в групу:**
```
1. Менеджер відкриває МЦ
2. Натискає "Перемістити в групу"
3. Обирає групу зі списку (дерево до 4 рівнів)
4. МЦ переміщується
```

**Навігація:**
- Дерево груп (collapsible)
- Клік на групу → показує всі МЦ в цій групі

## Донори та звітність

### Довідник донорів

**Хто створює:**
- Адміністратор
- Керівник

**Поля донора:**
- Назва організації (наприклад, "UNICEF", "Red Cross")
- Тип (Державна / Волонтерська / Міжнародна)
- Контакт (email, телефон)
- Шаблон звітності (PDF або Excel — буде додано пізніше)

### Прив'язка донора до МЦ

**При отриманні МЦ:**
- Донор вказується в **примітках** (необов'язково)

**При списанні МЦ:**
- Донор вказується **обов'язково** (якщо потрібна звітність)
- Вибір зі списку донорів

### Звітність для донорів

**Сценарій:**
МЦ списано (використано або втрачено) → потрібно звітувати донору.

**Workflow:**
```
1. Менеджер списує МЦ → вказує донора
2. Система генерує звіт (PDF або Excel) на основі шаблону донора
3. Звіт містить:
   - Назва МЦ
   - Серійний номер
   - Кількість списана
   - Дата списання
   - Причина (використано / втрачено)
   - Розбивка по донорах (якщо комплект з декількох донорів)
4. Менеджер може:
   - Роздрукувати (PDF)
   - Експортувати (Excel)
   - Відправити email (майбутня фіча)
```

### Комплекти з декількох донорів

**Сценарій:**
Комплект "Комп'ютер" складається з компонентів від різних донорів:
- Монітор (UNICEF)
- Системний блок (Red Cross)
- Клавіатура (UNICEF)

**При списанні комплекту:**
```
1. Менеджер списує комплект
2. Вибирає донорів: UNICEF, Red Cross
3. Система генерує 1 акт списання
4. В звіті розбивка:
   - UNICEF: 2 МЦ (Монітор, Клавіатура)
   - Red Cross: 1 МЦ (Системний блок)
5. Загальна кількість: 3 МЦ
6. Звіт відправляється обом донорам (один і той же)
```

## AI інтеграція

### Ollama (локальна LLM)

**Де запускається:**
- На Mac mini (розробка)
- На production сервері (після запуску)

**RAM:** 24 GB

**Моделі:**
1. **LLaMA 3.2 Vision 11B** (~7 GB)
   - Розпізнавання фото накладних (OCR)
   - Структурування даних

2. **Qwen2.5 14B** (~9 GB)
   - Парсинг Excel таблиць
   - Відповіді на складні запити керівника (Analytics Bot)

**Разом:** ~16 GB (залишок 8 GB для OS, PostgreSQL, Node.js)

### Розпізнавання фото накладних

**Workflow:**
```
1. Менеджер надсилає фото накладної в Telegram Import Bot
2. Бот відправляє фото в Ollama API (LLaMA Vision)
3. Ollama розпізнає:
   - Назви МЦ
   - Серійні номери
   - Кількість
   - (Примітки, якщо є)
4. Ollama повертає structured JSON:
   {
     "items": [
       {
         "name": "Монітор Dell P2422H",
         "serial": "ABC123456",
         "quantity": 5,
         "unit": "шт",
         "notes": ""
       }
     ]
   }
5. Бот запитує менеджера:
   "Статус МЦ?"
   [Державне] [Волонтерське]
6. Менеджер обирає → бот показує preview
7. Менеджер підтверджує → МЦ додаються на склад
```

### Парсинг Excel таблиць

**Workflow:**
```
1. Менеджер надсилає Excel файл в Telegram Import Bot
2. Бот завантажує файл → відправляє в Ollama API (Qwen)
3. Ollama розпізнає структуру таблиці:
   - Колонки (Назва, Серійний номер, Кількість, тощо)
   - Рядки (кожен рядок = 1 МЦ)
4. Ollama повертає structured JSON (як вище)
5. Бот запитує статус → менеджер обирає
6. Підтвердження → МЦ додаються
```

## Telegram боти

### 1. Import Bot (для Менеджерів і Адміністраторів)

**Призначення:**
Додавання МЦ через фото/Excel.

**Хто використовує:**
- Менеджери
- Адміністратори

**Команди:**
- `/start` — авторизація (прив'язка Telegram ID до користувача)

## Нефункціональні вимоги

### Продуктивність

**Desktop App:**
- Запуск за <3 секунди
- Відкриття списку МЦ (1000+ позицій) за <1 секунду
- Синхронізація (при online) — <5 секунд для 100 операцій

**Backend API:**
- Відповідь на запит <200ms (95-й перцентиль)
- Підтримка 15 одночасних користувачів
- Ollama: розпізнавання фото накладної <10 секунд

### Безпека та шифрування 🔐

#### Загальні принципи
- **Zero Trust Architecture** — не довіряємо нікому по замовчуванню
- **Defence in Depth** — кілька рівнів захисту
- **Least Privilege** — мінімальні права доступу для кожної ролі

#### Аутентифікація

**Первинна реєстрація:**
- Тільки Адміністратор/Керівник створює акаунти
- Email verification (опціонально для Production)
- Сильні паролі (мінімум 12 символів, верхній/нижній регістр, цифри, спецсимволи)

**Логін:**
- Email + пароль
- JWT токени (30 днів TTL)
- Refresh token (90 днів)
- Хешування паролів (bcrypt, cost 12)

**Multi-factor Authentication (MFA):**
- ⏸️ Майбутня фіча (фаза 6+)
- Telegram як другий фактор

#### Авторизація

**Role-based access control (RBAC):**
- Менеджер не бачить дані інших менеджерів
- Адміністратор бачить тільки підлеглих
- Керівник бачить все (read-only для складів)

**Перевірка прав на кожному запиті:**
- Backend перевіряє JWT токен
- Backend перевіряє role користувача
- Backend перевіряє доступ до конкретного ресурсу (МЦ, склад, звіт)

#### Шифрування передачі даних

**Transport Layer Security (TLS):**
- ✅ **HTTPS тільки!** (TLS 1.3)
- Всі з'єднання між Desktop App і Backend шифруються
- Всі з'єднання між Telegram Bots і Backend шифруються
- Самопідписні сертифікати на Development
- Let's Encrypt на Production

**WebSocket (для real-time notifications):**
- WSS (WebSocket Secure) — шифрування через TLS
- JWT в header для аутентифікації

**Чутливі дані в transit:**
- Паролі **НІКОЛИ** не передаються в plain text
- JWT токени передаються в Authorization header
- Refresh tokens зберігаються в HTTP-only cookies (захист від XSS)

#### Шифрування даних в спокої (at rest)

**Локальна база даних (SQLite на Desktop App):**
- ✅ **SQLCipher** — шифрування SQLite файлу
- AES-256 encryption
- Ключ шифрування зберігається в OS keychain:
  - Windows: Credential Manager
  - Mac: Keychain Access
- При втраті комп'ютера — дані нечитабельні без пароля користувача

**Центральна база даних (PostgreSQL):**
- ⚠️ **Шифрування диску** — LUKS (Linux) або BitLocker (Windows)
- Опціонально: pgcrypto для шифрування окремих стовпців (наприклад, примітки з чутливою інформацією)

**Резервні копії:**
- Шифрування backup файлів (AES-256)
- Ключ зберігається окремо від backup

#### Шифрування між локаціями (міжміський зв'язок)

**Проблема:**
Менеджери в різних містах (Київ, Львів, Дніпро) передають дані через Internet.

**Рішення (обов'язково):**

**1. VPN (найкращий варіант):**
- ✅ **Tailscale** (WireGuard-based, zero-config VPN)
- Всі Desktop Apps підключаються через Tailscale mesh network
- End-to-end шифрування між пристроями
- Навіть якщо Backend скомпрометований — трафік між менеджерами зашифрований

**Альтернатива (якщо не можна VPN):**
- CloudFlare Tunnel (Zero Trust Network Access)
- Private network без публічних IP

**2. TLS 1.3 (мінімум):**
- Обов'язково для всіх HTTP з'єднань
- Certificate pinning на Desktop App (захист від MITM attacks)

**3. End-to-End Encryption для особливо чутливих даних:**
- ⏸️ Майбутня фіча (якщо потрібно)
- Фото з примітками можуть шифруватись на клієнті перед відправкою
- Тільки менеджер з правильним ключем може розшифрувати

#### Захист від витоку даних

**Desktop App:**
- Automatic screen lock після 15 хв неактивності
- Логаут при закритті програми (опціональна налаштування)
- Заборона screenshots для чутливих екранів (Windows/Mac API)

**Telegram Bots:**
- Повідомлення з чутливою інформацією видаляються через 1 годину (Telegram API)
- Можливість `/clear` — видалити всю історію з ботом

**Логи:**
- Чутлива інформація (паролі, токени) **НІКОЛИ** не логується
- Логи зберігаються в безпечній директорії (доступ тільки для root/admin)

#### Захист від атак

**SQL Injection:**
- Parameterized queries (prepared statements)
- ORM (TypeORM / Prisma) автоматично захищає

**XSS (Cross-Site Scripting):**
- Не застосовується (Desktop App, не веб)
- Але: sanitization user input перед збереженням в БД

**CSRF (Cross-Site Request Forgery):**
- JWT токени в Authorization header (не cookies)
- CORS налаштування (тільки дозволені origins)

**MITM (Man-in-the-Middle):**
- TLS 1.3 (обов'язково)
- Certificate pinning на Desktop App
- VPN (Tailscale) для додаткового захисту

**Brute Force:**
- Rate limiting на Backend (максимум 5 спроб логіну за хвилину)
- Account lockout після 10 невдалих спроб (30 хв)

**DDoS:**
- CloudFlare для Production (якщо VPS)
- Rate limiting на API endpoints

#### Аудит та моніторинг

**Логування безпекових подій:**
- Всі спроби логіну (успішні / невдалі)
- Всі зміни ролей користувачів
- Всі операції з чутливими даними (списання МЦ, передача)
- Всі зміни в налаштуваннях безпеки

**Alerts:**
- 5+ невдалих спроб логіну → notification Керівнику (Telegram Monitoring Bot)
- Підозріла активність (наприклад, логін з нової локації) → notification
- Спроби несанкціонованого доступу → notification

**Audit trail:**
- Хто що зробив, коли, з якого IP
- Зберігається 1 рік
- Неможливо видалити (тільки read-only для Керівника)

#### Compliance (відповідність стандартам)

**GDPR (якщо застосовується):**
- ⏸️ Поки не релевантно (внутрішнє використання в Україні)
- Але: можливість видалити особисті дані користувача (право на забуття)

**ISO 27001 (інформаційна безпека):**
- ⏸️ Майбутня сертифікація (якщо проєкт масштабується)

**Українське законодавство:**
- Закон про захист персональних даних
- Збереження даних на territory України (якщо вимагається)

#### Резервне копіювання та відновлення

**Backup strategy:**
- **Щоденні бекапи** PostgreSQL (повні)
- **Щогодинні інкрементальні** бекапи (критичні операції)
- Зберігання 30 днів

**Encryption:**
- Всі бекапи шифруються (AES-256)
- Ключ зберігається в окремому безпечному місці

**Disaster Recovery:**
- RPO (Recovery Point Objective): 1 година
- RTO (Recovery Time Objective): 4 години
- Тестування відновлення раз на місяць

#### Навчання користувачів

**Onboarding:**
- Кожен новий менеджер проходить short training (15 хв)
- Правила безпеки: не ділитись паролями, не залишати комп'ютер без блокування

**Документація:**
- Security guidelines (PDF) — для всіх користувачів
- Що робити при витоку даних / втраті пристрою

**Підсумок розділу "Безпека":**

✅ **Шифрування в transit:** TLS 1.3 + VPN (Tailscale)  
✅ **Шифрування at rest:** SQLCipher (Desktop), pgcrypto (Server), disk encryption

## Глосарій

- **МЦ** — Матеріальні цінності (товари, обладнання, тощо)
- **BOM** — Bill of Materials (склад комплекту)
- **Донор** — організація, яка передає МЦ (державна/волонтерська)
- **Комплект** — набір МЦ, зібраних разом
- **Псевдонім (Nickname)** — альтернативна назва для групи МЦ (для голосових запитів)
- **Ollama** — локальна LLM для AI-задач
- **Sync** — синхронізація даних між Desktop app і сервером
